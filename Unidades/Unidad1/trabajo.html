Normalización de una Boleta Electrónica

Caso práctico aplicado a la empresa Little Caesars, donde se analiza una boleta real para identificar entidades, atributos, relaciones y transformar la información hasta llegar a la Tercera Forma Normal (3FN).

Boleta Original

Toca la imagen para ampliarla
Esta boleta electrónica fue utilizada como documento fuente para el proceso técnico de normalización de datos.
Se identificaron entidades principales como cliente, empresa, productos, detalle de venta y pago.
También se detectaron atributos repetidos, campos dependientes y estructuras no optimizadas.
El análisis permitió diseñar una base de datos ordenada, escalable y profesional aplicando 1FN, 2FN y 3FN.
Boleta

Tabla Sin Normalizar 0FN

Estado inicial de los datos extraídos directamente desde la boleta electrónica, donde toda la información se encuentra mezclada en una sola estructura.

⚠ Problemas Detectados
Redundancia:
Datos como cliente, fecha y forma de pago se repiten.
Información mezclada:
Venta, productos y cliente están en una sola tabla.
Dificultad de mantenimiento:
Cualquier cambio requiere editar múltiples registros.
Riesgo de errores:
Existe mayor posibilidad de inconsistencias.
Baja escalabilidad:
No es una estructura apta para crecer profesionalmente.

Primera Forma Normal (1FN)

En esta etapa se eliminan grupos repetitivos y cada campo pasa a contener un único valor. Se comienza a separar la información principal.

Segunda Forma Normal (2FN)

Se eliminan dependencias parciales separando productos y detalle de venta, mejorando la organización de la información.

Tercera Forma Normal (3FN)

En esta etapa final se eliminan dependencias transitivas y se obtiene una estructura sólida, organizada y profesional. Cada tabla cumple una función específica dentro del sistema, permitiendo escalabilidad, integridad y mejor rendimiento.
